# 基于 i.MX6UL 的嵌入式人脸识别门禁终端

在 i.MX6UL (Cortex-A7, 单核, 无 NPU) 平台部署**真实神经网络**模型。
使用 ncnn 推理引擎，ncnn 内部用汇编语言针对特定的 CPU（比如 ARM 架构）写了大量的加速指令（NEON 技术），能同时让多个数据一起计算。神经网络运行期间会有海量的临时数据。推理引擎能做到“内存复用”，防止你的嵌入式小板子因为内存耗尽而死机。

UltraFace：检测人脸，极其轻量的人脸检测模型。它的体积通常只有 几百 KB 到几 MB，能在低功耗设备上轻松跑到 100 帧/秒以上。它的输入是图片，输出的是坐标框。它做的事情是：“图片里 (x_1, y_1) 到 (x_2, y_2) 这个矩形区域是一张脸。”
camera输出的数据是YUYUV 640*480，但是 UltraFace 固定的输入数据大小是 320*240，（我们选取RGB24类型是为了内存和计算速度考虑）。所以需要对图像进行缩小（注意：不是裁剪！）一般缩小（或者放大）有两种方式
1、最近邻插值：如果是缩小，按固定步长均匀采样，丢弃大部分像素，导致像素丢失，会有锯齿状产生。如果是方法，就按照步长复制像素点，也会造成转换过后的图像有锯齿状。
2、双线性插值：由于我们是从640*480 ──> 320*240。所以像素点数量正好缩小4倍。双线性的做法是加权平均，拿周围 4 个源像素按距离比例混合：

源图像 (640×480)              目标图像 (320×240)
    A ──── B                      P
    │      │                     这个像素点 P 的颜色 =
    │   P  │                         A×70% + B×30% + C×30% + D×10%
    │      │                        （4 个邻居按距离加权）
    C ──── D

原始图片尺寸是640×480，ultraface的输入图像尺寸是320×240，所以需要经过双线性插值转化为320×240尺寸。输出就是两个表，
1、“可能性图” (Scores)：维度[4420, 2]。其中一个表是维度是4420×2，该表格为每个锚点生成一个2维的置信度向量（即“不是人脸”和“是人脸”的概率）。表示每个锚点的置信度，置信度太小直接不予考虑，
2、“修正量图” (Boxes)：维度[4420, 4]。另一个表维度是 4420×4，表示每个锚点的修正量，4420个锚点，各生成4个修正参数（[dx, dy, dw, dh]），用于修正锚点的位置和大小。比如boxes[1500]  = [0.05, -0.03, 0.12, 0.10]，表示1500号锚点具体的修正。

UltraFace的4420个锚点，其实这4420并不是将图像均匀的分成4420份，而是分成4层。
第一层是stride 8层，划分出来了1200个格子，每个格子尺寸是40×30，并且每个格子有三种尺寸，这就是3600个锚点，
第二层是stride 16层，划分出来了300个格子，每个格子尺寸是20×15，并且每个格子有2种尺寸，这就是600个锚点，
第三层是stride 32层，划分出来了80个格子，每个格子尺寸是10×8，并且每个格子有2种尺寸，这就是160个锚点，
第四层是stride 64层，划分出来了20个格子，每个格子尺寸是5×4，并且每个格子有3种尺寸，这就是60个锚点，
所以最后就是3600 + 600 + 160 + 60 =4420个锚点，这种分层主要为了检测不同远近的脸（因为远距离人脸比较小，近距离人脸比较大）。
格子是锚点的中心位置，但是每个格子不是只能放一个框，每个格子有几种尺寸就有几种框。
比如：stride=8层, 格子坐标 (20, 12)
    同一个中心点 ●，有三种尺寸24px、16px、10px
        ┌──────┐  ← 24px 锚点 (检测偏大的脸)
        │ ┌──┐ │  ← 16px 锚点
        │ │● │ │  ← 10px 锚点 (检测偏小的脸)
        │ └──┘ │
        └──────┘
三种尺寸都从同一点出发，各自独立参与后续的筛选。哪个和真实人脸的实际尺寸最接近，哪个的 scores 就最高。
UltraFace模型最后只会吐出Scores来表示每个锚点的置信度，以及Boxes表示每个锚点的偏移量，然后我们再根据偏移量在程序中计算实际人脸框。
伪代码如下：
    for i = 0 to 4419:
        if scores[i] 中"是脸"份数太低 → 跳过
        取出 boxes[i] 的 4 个偏移值
        套入修正公式 → 算出实际像素坐标
    NMS 去重 → 输出最终人脸框

UltraFace模型最后得到的Scores和Boxes可以计算出很多由锚点转化而成的人脸框，所以最后需要根据通过NMS（非极大值抑制，Non-Maximum Suppression）筛选到只剩下一个人脸框。具体步骤如下：
1、排序：把所有候选框按照“是人脸的概率”从大到小排个序，先挑出概率最高的那一个（绝对是人脸的框），把它保护起来。
2、算重叠度：拿着这个概率最高的框，去和剩下所有的框对比，计算它们的交并比。
3、砍掉重叠框：如果发现别的框和这个最高概率框的重叠度太高（比如IoU >0.45），说明它们圈的是同一个人脸，属于“重复多余的框”，直接把这些低概率的重叠框无情抹杀（抑制）
  图示：
       A 框              
    ┌─────────┐            
    │  ┌──────│──┐         
    │  │ 交集  │  │    IoU = 交集面积 / (A面积 + B面积 - 交集面积)
    └──│──────┘  │         = 0.6  → 重叠 60%，同一张脸
       └─────────┘         = 0.05 → 基本不重叠，两张不同的脸

在utils.c文件中的rgb24_crop_resize()函数有两个作用：
1. crop（裁剪）：从 640×480 大图里把 UltraFace 输出的矩形抠出来
2. resize（缩放）：不管抠出来的脸是 200×260 还是 50×60，统一用最近邻采样缩到 112×112，因为 MobileFaceNet 模型需要输入 112×112 的人脸图像
  
MobileFaceNet：人脸识别，针对移动端优化的人脸识别模型。他的输入是UltraFace 裁剪出来的那张人脸小图。而输出是一个 [1*128] 维度的浮点数特征向量，ncnn会把 UltraFace 返回的人脸区域缩放到模型要求的尺寸（比如 112x112 像素），并把像素值（0~255）归一化（变成 0.0 ~ 1.0 之间的浮点数）。然后将这组像素浮点数作为第一层的输入，乘以第一层的权重，结果送到第二层……以此类推。最后一层网络计算结束，吐出来的不再是像素，而是一个包含了 128 个浮点数的数组（即 [1*128] 向量）。

欧氏距离比对实现身份验证。dist = √[(a₁-b₁)² + (a₂-b₂)² + ... + (a₁₂₈-b₁₂₈)²]

训练好的模型（UltraFace、MobileFaceNet），也叫做“预训练模型”。本身只是一堆权重参数和网络结构描述，不能直接运行。推理引擎的作用是：读取模型文件（如 .param、.bin、.onnx）。如果用记事本强行打开一个模型文件（如 .bin、.onnx、.param），你会看到一堆乱码。但本质上，它里面只存放了两样东西：
1、网络的“骨架”（拓扑结构）：它记录了模型一共有多少层（Layer），每一层是什么类型（如卷积层、池化层），以及数据流向的先后顺序。在 ncnn 中，这通常存放在 .param 文件中。
2、网络的“血肉”（权重参数 Weights）：它记录了训练完成后，每一个节点上精确的数字。在 ncnn 中，这存放在 .bin 文件中。


UltraFace 和 MobileFaceNet 理论上可以接收RGB24，RGB32甚至是灰度图像，但是RGB24每个像素字需要3个节数，而RGB32每个像素字需要4个节数（R, G, B三个颜色通道各一个字节+ Alpha (A) 透明度通道的一个字节），所以RGB24内存占用较少，计算速度也更快，像素归一化是为了防止数值过大导致网络计算炸掉，UltraFace 或 MobileFaceNet 他们归一化的区间都是[-1，1]。
归一化是需要再推理之前进行的，而推理之后就会得到一个[1,128]维度的输出向量。
如果两张同一个人的人脸照片，一张光线亮一张暗，提取出的原始向量数值差异很大，那么直接算这两张照片的欧氏距离就会很大，最后判定这是两个人，归一化后只剩方向差异（光线影响被消除），欧氏距离才准确。
framebuffer接收的数据必须是RGB32。
camera捕获到的图像是YUYV 640×480
UltraFace的输入是RGB24 320×240

怎么将参考人脸转化成128维向量和mobilefacenet得到的特征向量进行对比的呢？
1、将参考人脸怎么变成 128 维向量：在 PC 端写了一个 Python 脚本 gen_reference.py。它先用 OpenCV 的 Haar Cascade 在标准照上检测出人脸区域，裁剪后缩放到 112×112，然后调用 MobileFaceNet 的 ONNX 模型（通过onnxruntime）做一次推理，输出一个 128 维的浮点数组（为什么是128维而不是256维：维度越高越精确，但是计算量翻倍，存储也翻倍，并且边际收益递减，128维是移动端性能和精度的最佳平衡点）。最后做 L2 归一化，把 128 个 float 存成 512 字节（特征向量通常以 32 位浮点数（float32）存储，每个浮点占 4 字节，所以二进制文件大小就是4 * 128 = 512）的 reference.bin文件（因为.bin二进制文件可以直接通过fread进行读取），推到嵌入式板子上。
2、板端对比：板端程序启动时用 fread 把这 128 个 float 读进内存里的 ref_embedding[128] 数组。之后每帧摄像头画面经过 UltraFace 检测人脸 → 裁剪 112×112 → MobileFaceNet 推理 → 得到实时 embedding[128]，然后调 face_recog_compare() 算两个向量的欧氏距离。


在模型训练阶段，拿出 10,000 张照片喂给神经网络。不断微调神经网络的权重系数，这个过程叫做反向传播。训练结束后的成果，就是那张写满了最终完美系数的表格（即 .onnx 文件）。在PC上使用 onnx2ncnn 工具，将训练好的 .onnx 模型转换为 ncnn 需要的 .param 和 .bin 文件，然后把这些文件放到你的开发板上，由主程序（C/C++代码）调用 ncnn 库来加载和执行。

## 推理引擎

**[ncnn](https://github.com/Tencent/ncnn)** — 腾讯开源, 专为 ARM NEON 优化的神经网络推理框架。
- 纯 C++ 实现, 零系统依赖
- Cortex-A7 NEON 加速, 预计 3~8 FPS 全流程
- 支持 Vulkan/OpenMP 加速 (i.MX6UL 上不用)

## 编译

### 1. 交叉编译 ncnn

```bash
git clone https://github.com/Tencent/ncnn.git
cd ncnn && mkdir build-arm && cd build-arm

cmake -DCMAKE_TOOLCHAIN_FILE=../toolchains/arm-linux-gnueabihf.toolchain.cmake \
      -DNCNN_BUILD_TOOLS=OFF -DNCNN_BUILD_EXAMPLES=OFF \
      -DCMAKE_INSTALL_PREFIX=/path/to/ncnn-install ..

make -j4 && make install
```

### 2. 准备模型文件

```bash
cd scripts && ./prepare_ncnn_models.sh
```

该脚本会:
- 下载 UltraFace ONNX → 转为 ncnn 格式
- 将 MobileFaceNet ONNX → 转为 ncnn 格式
- 输出 `models/*.param` + `models/*.bin`

### 3. 编译项目

```bash
# 编辑 Makefile, 设置 ncnn 路径:
#   NCNN_INC = /path/to/ncnn-install/include/ncnn
#   NCNN_LIB = /path/to/ncnn-install/lib

# Mock 模式 (x86, 无硬件/无模型, 逻辑测试)
make mock CC=gcc CXX=g++

# 交叉编译 (i.MX6UL)
make
```

### 4. 部署运行

```bash
make run
```
