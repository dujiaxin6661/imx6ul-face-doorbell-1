嵌入式板子（比如你 i.MX6ULL）内存和存储空间非常有限，所以官方或前人做系统时，会把不需要的外设驱动（比如板子上没插网卡时，就把 Wi-Fi 驱动）全部裁剪掉，以此来减小内核体积、加快开机速度。

关于 make menuconfig：它就是内核的图形化配置工具。
1、[*] (Built-in)：直接编译进内核镜像中，必须把重新编译出来的庞大内核文件重新烧写到板子的 Flash 闪存里，之后开机就自带。
2、<M> (Module)：原本的内核不动，编译成独立的 .ko 驱动文件（模块），像插件一样，需要时随时手动加载，不占内核开机空间。
3、[ ] (Empty)：完全不要，彻底裁剪。

make ARCH=arm CROSS_COMPILE=arm-buildroot-linux-gnueabihf- modules
这条命令的作用是：使用指定的 ARM 交叉编译工具链，只编译 Linux 内核中配置为模块（M）的部分，而不重新编译整个内核镜像（zImage/uImage）。

最后将编译出来8723bu.ko 驱动文件，通过 U 盘拷贝到/lib/modules/ 目录下。然后用 modprobe 动态加载它。

modprobe 是 Linux 中用于加载内核模块的命令，相比 insmod 它更智能、更常用。
1、自动处理依赖：当你要加载模块 A，而 A 依赖模块 B 时，modprobe 会先加载 B，再加载 A。insmod 不会处理依赖，需要手动按顺序加载。
2、从标准路径加载：默认从 /lib/modules/$(uname -r)/ 下搜索模块。而 insmod 必须指定模块文件的完整路径。

“因为嵌入式系统追求轻量化，板子原厂的内核默认裁剪掉了无线网络协议栈和 RTL8723BU 的驱动。
我通过 make menuconfig 修复了依赖关系，并将该驱动配置为了内核模块（<M>）。
随后，我利用交叉工具链**仅编译出了该外设的 .ko 驱动模块**，并通过 U 盘这种离线方式，将 .ko` 文件和固件动态部署到了板子的文件系统目录中。
这种‘编译为模块、动态加载’的方案，避免了重新烧写整块板子内核的风险，实现了外设驱动的无缝升级。”
